# DevForge 全局规则

## 语言与沟通

- 使用中文与用户交流。
- 所有源文件使用 UTF-8 without BOM 编码。
- 中文文本禁止转义为 `\uXXXX`。
- 仅在解释非显而易见的意图、约束或风险时使用中文注释。
- 标识符、API 名称、数据库字段和 commit type 使用英文。

## 项目边界

- DevForge 是 Windows-first、本地优先的开发者知识与 AI 工作台。
- Rust 核心必须保持可移植到 macOS 和 Linux。
- Tauri 是桌面宿主和 IPC 适配器，不是业务逻辑层。
- React 是展示层，不得被视为安全边界。
- SQLite 是业务元数据的唯一事实来源。
- Tantivy、向量索引、摘要和 Code Graph 索引是可重建数据。
- AI 生成的内容不是项目事实，除非有源证据支持或用户明确确认。

## 架构规则

- Domain 代码不得依赖 Tauri、React、SQLite、HTTP Client、操作系统 API 或模型 SDK。
- Application Service 依赖 Trait，不依赖具体基础设施实现。
- 基础设施代码实现 Application 定义的 Port 和 Adapter。
- Tauri Command 暴露业务用例，不得暴露数据库式 CRUD。
- React 组件不得直接调用 `invoke()`，所有 IPC 访问通过共享 API Client。
- 平台特定行为必须隔离在 Platform Adapter 中。
- 不引入没有当前具体需求的新 crate、package、抽象或 service。
- 优先使用职责单一的聚焦模块，而非大型多功能文件。

## 范围纪律

- 开始修改前，阅读当前阶段规格和已批准的实施计划。
- 只实现当前分配的任务，不提前实现后续阶段、相邻功能或投机性扩展点。
- 不执行与任务无关的重构。
- 不得静默更改已批准的架构决策。
- 实施中发现需要变更已批准设计时，停止并说明冲突后再继续。
- 已接受的架构变更记录到相关设计文档或 ADR 中。

## 工作原则

- 修改行为前先写失败测试。
- 实现通过测试所需的最小代码。
- 完成后运行相关测试、格式化、Lint 和类型检查。
- 审查完整 diff，确认无非预期变更。
- 未运行验证时，不得声称功能已经完成。

## 验证底线

- 不得仅基于代码检查就声称测试通过。
- 报告时必须包含实际执行的命令及其结果。
- 区分：验证成功 / 未运行 / 当前环境不可用 / 因既有原因失败。
- 相关测试、格式化、Lint 或类型检查未通过时，任务不算完成。
- 不得为通过构建而隐藏、删除、跳过、削弱或重写失败测试。

## 工程默认值

- Safe Rust 是默认选择；`unsafe` 代码必须隔离并附带 SAFETY 不变量说明。
- 跨 crate、Trait、IPC 或 Application 边界的错误使用 `thiserror`。
- `anyhow` 只允许用于二进制入口、启动编排、迁移工具和诊断程序。
- 优先使用函数、Trait、enum 和 builder，避免自定义宏。
- 不得在未评估维护状态、许可证、unsafe 使用、Windows 支持、编译成本和传递依赖的情况下添加依赖。
- TanStack Query 管理 Rust 持久化数据；Zustand 只管理共享瞬态 UI 状态。
- `useEffect` 用于同步外部系统，不用于派生或复制状态。
- Tokio 是唯一的异步运行时。
- 长期任务必须可监管、可取消、可观测、可恢复。
- 所有外部输入按不可信数据处理，运行时校验。
