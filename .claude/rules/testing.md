# 测试与验证规则

## 测试原则

- 测试外部可观察的行为和不变量，不测试私有实现细节。
- 每个 bug 修复始于修复前会失败的回归测试。
- Domain 状态机需要表驱动的转换测试。
- Repository 实现需要真实临时数据库测试。
- Provider、连接器、存储和执行适配器需要契约测试。
- 不得在单元测试中访问真实外部网络。
- 使用确定性时钟、标识符、随机种子和临时路径。
- 不得添加任意 sleep 使异步测试通过，使用显式同步或 fake time。
- 安全边界必须通过直接调用 Rust API 进行测试，不仅通过 UI。
- 只有在需求变更时才可移除或削弱测试，且必须说明变更原因。

## Fixture 规则

- Fixture 数据必须确定性且可重现。
- Fixture 必须版本化，变更需要记录。
- 不得在 fixture 中硬编码环境特定路径或凭据。
- 大型 fixture 应使用文件存储，不内联在测试代码中。
- Fixture 应覆盖正常路径、边界条件和已知错误场景。
- 索引器、解析器和搜索回归数据使用 `fixtures/` 目录。

## Property Test

- 路径规范化、Patch 解析和状态转换需要属性测试。
- 属性测试应覆盖：往返不变量、幂等性、单调性、边界收缩。
- 使用 `proptest` 作为 Rust 属性测试框架。
- 属性测试的最小用例必须可重现（记录 seed）。
- 不得用属性测试替代关键路径的确定性单元测试。

## Fuzz 测试

- 模型输入、Webhook 输入、Patch 输入和插件输入的解析器应该有 fuzz 测试。
- 使用 `cargo-fuzz` 作为 Rust fuzz 框架。
- Fuzz 测试发现的每个崩溃都必须有对应的回归测试。
- Fuzz 目标必须能在合理时间内发现有意义的覆盖率。
- 不得将 fuzz 测试作为唯一的安全验证手段。

## Snapshot 测试

- 快照测试仅适用于稳定的结构化输出。
- 不得使用快照替代语义断言。
- 快照文件变更必须在 code review 中人工确认。
- 不得自动更新快照而不检查变更原因。
- 快照应包含足够的上下文以理解预期输出。

## Benchmark 规则

- Benchmark 必须度量一个命名的性能假设。
- 记录 fixture 大小和机器相关的假设条件。
- 不得将正确性断言与性能阈值混在一起。
- 声称优化前必须与基线对比。
- 不得仅因为微基准测试改善了就优化代码。
- 性能变更必须包含与真实 DevForge 工作流相关的内存、延迟或吞吐量证据。
- Benchmark 结果必须可重现，记录环境条件。
- 不得在 CI 中使用不稳定的 benchmark 结果阻塞合并。
