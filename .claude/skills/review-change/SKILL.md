# Review Change

## 核心原则

审查的目标是发现真实问题，不是总结优点。重点关注 bug、行为回归、安全问题、数据一致性、并发/异步问题、边界条件、错误处理和缺失测试。

## 1. 理解变更范围

- 运行 `git diff --name-only` 和 `git diff --stat` 确认变更范围。
- 阅读变更涉及的设计文档或 ADR。
- 理解变更的业务目标和技术背景。
- 确认变更与已批准计划的一致性。

## 2. 需求正确性

检查：

- 变更是否实现了计划中声明的目标。
- 是否有计划中要求但未实现的部分。
- 是否有计划中未要求但额外实现的部分（范围蔓延）。
- 验收标准是否全部满足。
- 非目标是否被遵守。

## 3. 架构合规

对照 `always.md` 和对应路径规则检查：

- Domain 代码是否依赖了 Tauri、React、SQLite、HTTP Client 或模型 SDK。
- Application Service 是否依赖 Trait 而非具体实现。
- Tauri Command 是否暴露用例而非 CRUD。
- React 是否直接调用了 `invoke()`。
- 新增的 crate、package 或抽象是否有当前具体需求。
- 可见性是否尽可能窄。

## 4. 安全审查

检查：

- 路径是否在权限检查前规范化。
- 是否存在路径遍历、符号链接逃逸风险。
- 凭据是否以明文存储或出现在日志中。
- 审批是否绑定到精确参数和内容哈希。
- 外部输入是否按不可信数据处理。
- 命令执行是否经过风险分级。
- Prompt 注入防护是否到位。

## 5. 并发与异步

检查：

- 是否有散落的 `tokio::spawn` 未注册到 Supervisor。
- 是否有锁跨越 `.await`。
- Channel 是否有界。
- 长期任务是否支持取消和超时。
- 外部操作是否有超时。
- CPU 密集操作是否阻塞了 Tokio worker 线程。
- 是否有未处理的取消传播。

## 6. 错误处理

检查：

- 跨边界错误是否使用 `thiserror`。
- 是否有 `anyhow` 泄漏到公共 API。
- 是否有 `unwrap()`、`expect()` 在生产路径中。
- 错误是否在每一层重复记录。
- 错误变体是否可操作且语义可区分。
- 原始错误源是否被保留。

## 7. 边界条件

检查：

- 空集合、空字符串、零值的处理。
- 整数溢出、浮点精度。
- Unicode 和编码问题。
- 文件系统边界（路径长度、权限、不存在）。
- 并发访问同一资源。
- 最大值和最小值场景。
- 状态机的非法转换。

## 8. 测试覆盖

检查：

- 新行为是否有测试。
- Bug 修复是否有回归测试（修复前会失败）。
- 边界条件是否有测试。
- 错误路径是否有测试。
- 安全边界是否通过 Rust API 直接测试。
- 测试是否使用确定性数据。
- 是否有测试依赖外部网络或环境。

## 9. 前端特定审查

当变更涉及 React/TypeScript 时额外检查：

- 状态归属是否正确（Query 管服务端数据，Zustand 管 UI）。
- 是否有 `useEffect` 用于派生或复制状态。
- 是否将 Query 数据复制到了 Zustand。
- 高频流是否使用了批量更新和有界窗口。
- 大型列表是否使用了虚拟化。
- TypeScript 是否启用了 strict。
- 是否有 `any`、类型断言或非空断言绕过类型系统。

## 10. 依赖审查

当变更引入新依赖时检查：

- 是否评估了维护状态、许可证、unsafe 使用、Windows 支持。
- 是否有现有依赖可替代。
- Feature 是否最小化。
- Lockfile 变更是否只包含预期操作。
- 是否混入了无关的依赖升级。

## 11. 报告审查结果

### 发现问题格式

每个问题包含：

- **文件与行号** — 精确定位。
- **问题描述** — 一句话说明。
- **触发条件** — 什么情况下会触发。
- **影响** — 严重程度（P0/P1/P2/P3）。
- **建议修复** — 具体的修复方向。

### 严重程度定义

- **P0** — 安全漏洞、数据丢失、崩溃。必须在合并前修复。
- **P1** — 行为回归、正确性问题、性能严重退化。强烈建议修复。
- **P2** — 边界条件、可维护性风险、测试缺口。建议修复。
- **P3** — 风格问题、文档缺失。可选修复。

### 汇总结构

```markdown
## 审查结论

- P0: N 个
- P1: N 个
- P2: N 个
- P3: N 个

## 详细发现

### [P0] 问题标题

- **文件**: `path/to/file.rs:42`
- **描述**: ...
- **触发条件**: ...
- **影响**: ...
- **建议修复**: ...

## 未发现明确问题的领域

列出已审查但未发现问题的维度，以及仍未验证的风险。
```

### 无问题时

如果未发现问题，明确说明：

- 已审查的维度。
- 未发现明确问题。
- 仍未验证的风险（如需要运行时验证、外部服务测试等）。

不得仅说"代码看起来合理"就跳过边界条件分析。
