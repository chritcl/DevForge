# Implement Task

## 核心原则

每次只实现一个独立可测试的任务。先定义失败验证，再实现使其通过。

根据任务类型选择验证方式：

- **行为功能和 Bug 修复** — 先写失败测试（单元测试、集成测试、契约测试）。
- **构建、配置、脚手架和文档** — 先定义可执行的失败验证（构建命令、配置检查、文件结构断言），再实施并验证通过。

## 1. 理解任务

- 阅读当前阶段规格和已批准的实施计划。
- 确认本次任务的目标、约束和验收标准。
- 检查 `git status --short`，保护用户已有修改。
- 明确非目标：不提前实现后续任务、相邻功能或投机性扩展点。
- 不执行与任务无关的重构。

## 2. 探索现有代码

- 阅读任务涉及的现有文件和调用链。
- 理解现有的架构、命名、风格和工具链。
- 确认任务依赖的前置代码是否已存在。
- 如发现计划与现有代码冲突，暂停并说明冲突。

## 3. 定义失败验证

根据任务类型选择验证方式。

### 行为功能和 Bug 修复

- 确定最小的独立可测试交付物。
- 编写覆盖任务验收标准的测试。
- 运行测试，确认它**因预期原因失败**。
- 测试应覆盖：
  - 正常路径
  - 边界条件
  - 已知错误场景
- Domain 状态转换使用表驱动测试。
- Repository 实现使用真实临时数据库测试。

### 构建、配置、脚手架和文档

- 定义可执行的失败验证，例如：
  - `cargo build --workspace` 失败（缺少 crate 或配置）
  - `pnpm typecheck` 失败（缺少 tsconfig 或类型定义）
  - 文件结构断言（预期文件不存在）
  - 配置检查命令失败（ESLint、CI 配置等）
  - 文档链接或结构校验失败
- 运行验证命令，确认它**因预期原因失败**。
- 不要求对配置和脚手架代码写单元测试，但必须有可执行的验证手段。

## 4. 实现最小代码

- 实现通过测试所需的最小代码。
- 遵循项目已有架构、命名、风格和工具链。
- 遵循 `.claude/rules/` 中对应路径的规则文件。
- 优先借用而非克隆。
- 可见性尽可能窄。
- 不引入没有当前需求的抽象。

## 5. 运行验证

按以下顺序执行，每步记录命令和结果：

1. **聚焦测试** — 运行本次任务直接相关的测试。
2. **格式化** — `cargo fmt --check` 或 `pnpm exec prettier --check`。
3. **Lint** — `cargo clippy` 或 `pnpm lint`。
4. **类型检查** — `pnpm typecheck`。
5. **相关模块测试** — 运行受影响模块的完整测试套件。

任何步骤失败时：

- 立即修复，不得跳过。
- 不得隐藏、删除、跳过、削弱或重写失败测试。
- 修复后重新运行该步骤。

## 6. 审查变更

审查完整 diff，确认：

- 只有任务预期的变更，无非预期修改。
- 没有引入未在计划中的依赖。
- 没有引入未评估的新 crate 或 package。
- 错误处理符合规则（thiserror 跨边界，anyhow 只在顶层）。
- 没有 `unwrap()`、`expect()` 在生产路径中。
- 没有散落的 `tokio::spawn`。
- 没有锁跨越 `.await`。
- 注释只在必要时使用中文。

## 7. 报告结果

按以下结构汇报：

### 完成内容

说明实现了什么，对应计划中的哪一项。

### 变更文件

列出所有修改的文件和变更摘要。

### 验证结果

| 命令 | 结果 |
|------|------|
| 测试 | ✅ 通过 / ❌ 失败（附输出） |
| 格式化 | ✅ / ❌ |
| Lint | ✅ / ❌ |
| 类型检查 | ✅ / ❌ |

### 未验证项

列出无法在当前环境验证的项目及原因。

### 剩余风险

说明已知风险和后续关注点。

## 8. 多任务执行

当计划包含多个任务时：

- 每个任务独立执行步骤 1-7。
- 每个任务完成后汇报结果。
- 下一个任务开始前确认上一个任务的验证通过。
- 不得跳过中间任务直接实现后续任务。
- 不得合并多个任务为一次实现。
