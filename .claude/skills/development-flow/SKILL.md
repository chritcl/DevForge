# Development Flow

## 目标

由主智能体保持需求、关键决策和最终结果；将搜索、日志、实现细节、审查和验证委派给边界清晰的子智能体，降低主上下文污染并提高可验证性。

## 1. 建立任务边界

- 重述目标、约束和验收标准。
- 检查 `git status --short`，保护用户已有修改。
- 明确本次非目标。
- 将任务分为小型、中型或大型。
- 小型任务可以直接完成，但仍必须验证。
- 中大型任务继续以下流程。

## 2. 探索代码库

调用 `repository-explorer`。

存在多个独立问题时，并行启动多个探索任务，例如：

- 入口与调用链。
- 状态管理和数据流。
- 测试与构建入口。
- Rust/Tauri IPC 或后端接口。
- 历史实现和兼容性约束。

每个探索任务必须有不同调查边界。不要让多个智能体重复扫描整个仓库。

主智能体汇总：

- 已证实事实。
- 推断。
- 未知项。
- 需要设计解决的问题。

## 3. 设计与计划

调用 `solution-architect`，要求输出：

- 目标与非目标。
- 推荐方案。
- 工作项和依赖关系。
- 文件所有权。
- 可并行与必须串行的部分。
- 测试与验收标准。
- 风险与回滚方式。

主智能体检查计划：

- 是否存在两个实现者修改同一文件。
- 是否依赖未提交代码。
- 是否包含无关重构。
- 是否有明确完成标准。
- 是否能在单次开发周期内完成。

## 4. 选择实现方式

### 默认方式

使用 `implementer` 在当前工作区串行实现。适用于：

- 依赖当前分支或未提交修改。
- 工作项存在依赖。
- 多个工作项会修改相同文件。
- 非 Git 仓库。
- 不确定 worktree 基线是否包含所需代码。

### 隔离并行方式

仅在以下条件全部满足时使用 `isolated-implementer`：

- 相关基线代码已经提交并能在独立 worktree 中看到。
- 工作项完全独立。
- 文件所有权不重叠。
- 不依赖主工作区未提交修改。
- 合并结果可低冲突集成。

否则不要为了“看起来并行”而并行写代码。

## 5. 实现

为每个实现任务提供：

- 单一目标。
- 必要代码证据。
- 允许修改的文件。
- 禁止修改的文件。
- 验收标准。
- 必须运行的最小验证。

实现过程中：

- 优先添加回归测试。
- 保持改动最小。
- 每完成一个逻辑单元就执行相关验证。
- 发现计划不成立时暂停并重新规划。
- 遇到失败或异常行为，加载 `systematic-debugging` Skill，必要时调用 `debugger`。

## 6. 独立审查与验证

实现完成后并行调用：

### code-reviewer

检查：

- 需求正确性。
- 安全与数据风险。
- 边界条件和回归。
- 兼容性。
- 测试缺口。

### verifier

执行：

- `git diff --check`
- 相关测试
- 类型检查
- Lint
- 格式检查
- Rust 检查与测试
- 构建
- 必要运行验证

两者职责不得互相替代。

## 7. 修复循环

存在 P0/P1、明确 P2 或验证失败时：

1. 将具体问题和证据交给 `implementer` 或 `debugger`。
2. 只修复已确认问题。
3. 重新调用 `code-reviewer` 和 `verifier`。
4. 最多重复三轮。

连续两轮没有实质进展时停止循环，报告：

- 当前阻塞。
- 已验证假设。
- 失败证据。
- 需要用户或外部环境提供的内容。

## 8. 完成门槛

只有同时满足以下条件才宣布完成：

- 需求已经实现。
- 相关测试有实际结果。
- 静态检查和构建已执行，或未执行原因已明确。
- 没有未处理的 P0/P1。
- 无法验证的部分已披露。
- 没有覆盖用户已有修改。
- 没有未经授权的提交、推送或依赖变更。

## 9. 最终输出

按以下结构汇总：

### 完成内容

### 核心修改文件

### 验证结果

列出命令和结果。

### 未验证项

### 剩余风险

不要粘贴大段子智能体原始日志，只保留结论和关键证据。
