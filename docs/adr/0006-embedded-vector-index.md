# ADR-0006：使用嵌入式向量索引

## 状态

Accepted

## 背景

DevForge 需要支持语义搜索，让开发者可以用自然语言查找代码和文档。需要选择向量索引方案。

## 决策

使用嵌入式向量索引（每 Embedding 模型独立目录），不依赖外部向量数据库服务。

## 备选方案

- 嵌入式向量索引（推荐）
- 独立 Milvus / Qdrant / Weaviate 服务
- SQLite 扩展（sqlite-vss）
- PostgreSQL + pgvector

## 原因

- 桌面应用不应要求用户额外安装向量数据库服务
- 嵌入式方案零配置，开箱即用
- 每个 Embedding 模型使用独立索引目录，不同模型的向量不混用
- 模型切换时旧索引继续可用，新索引后台渐进构建
- 本地运行，向量数据不离开用户设备
- 索引属于可重建数据，损坏时可从 SQLite 重新生成

## 后果

- 需要自行管理向量索引的版本和一致性
- 切换 Embedding 模型时需要后台重新生成向量
- 每条向量记录必须包含 embedding_model_id，防止跨模型比较
- 向量生成失败不应阻塞全文索引的可用性
- 第一版规模有限，大型 Monorepo 可能需要后续优化

## 以后何时重新评估

- 当本地嵌入式方案无法满足百万级 Chunk 的检索性能时
- 当需要支持多设备向量同步时
- 当 SQLite 向量扩展成熟到可以替代独立方案时
- 当用户明确需要连接外部向量数据库时
