# ADR-0009：使用 WASM 作为插件运行时

## 状态

Accepted

## 背景

DevForge 需要一个可扩展的插件系统，允许第三方开发者添加连接器、解析器和只读工具，同时保证安全隔离。

## 决策

采用 **内置核心模块 + WASM 扩展插件 + 后续可选子进程插件** 的方案。第一版使用 Wasmtime 作为 WASM 运行时。

## 备选方案

- Rust 原生动态库插件
- 独立子进程插件
- WebAssembly Component 插件（推荐）
- JavaScript/TypeScript 插件

## 原因

- WASM 跨平台，插件无需为每个操作系统单独编译
- 接口明确，通过 Component Model 描述组件与宿主之间的能力
- 不直接共享宿主内存，安全性较高
- 可控制暴露给插件的宿主能力
- 可设置执行时间和资源限制
- 插件崩溃通常不会直接崩溃主应用
- Wasmtime 支持 WASI，可选择性暴露系统接口

## 后果

- WASM 生态仍不如普通原生插件成熟
- 复杂语言运行时可能增大插件体积
- 不适合直接构建复杂 React 页面
- 高性能解析器仍优先内置 Rust 实现
- 需要定义 WIT 接口和 Host Capability
- 插件权限需要显式声明和用户批准
- 第一版不允许第三方插件提供命令执行和 UI 注入能力

## 以后何时重新评估

- 当 WASM Component Model 生态成熟到可以替代当前方案时
- 当插件性能成为瓶颈，需要原生插件方案时
- 当需要支持 UI 插件时，可能需要额外的安全沙箱方案
- 当团队决定支持任意语言编写插件时
