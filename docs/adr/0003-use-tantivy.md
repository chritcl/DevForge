# ADR-0003：使用 Tantivy 作为全文索引

## 状态

Accepted

## 背景

需要支持代码、路径、符号和文档的本地全文搜索。

## 决策

使用 Tantivy；SQLite 继续作为业务数据事实来源。

## 备选方案

- SQLite FTS5
- 独立 Elasticsearch
- Meilisearch

## 原因

- Tantivy 是纯 Rust 库，可直接嵌入桌面应用，无需额外服务进程
- 支持自定义 Tokenizer，适合代码的驼峰和下划线分词
- 支持字段权重，可以区分符号名、标题、路径和正文的重要性
- 每个工作区使用独立索引目录，便于单独重建和删除
- 性能优于 SQLite FTS5，尤其在大规模代码库场景
- Elasticsearch 和 Meilisearch 需要独立服务进程，不适合本地桌面应用

## 后果

- 搜索索引属于可重建数据
- 需要维护 SQLite 和 Tantivy 的一致性
- 每个工作区使用独立索引目录
- 支持自定义 Tokenizer（代码驼峰、下划线分词）
- 支持字段权重（符号名 > 标题 > 路径 > 正文）

## 以后何时重新评估

- 当 Tantivy 不再维护或出现重大兼容性问题时
- 当需要分布式搜索能力时
- 当 SQLite FTS5 性能足以覆盖使用场景时
- 当需要更复杂的全文搜索特性（如同义词、模糊匹配高级规则）时
