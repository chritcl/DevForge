# DevForge 产品愿景

## 产品定位

> 面向开发者的本地知识库、跨项目代码检索与受控 AI 编程工作台。

DevForge 不是一个简单的 AI 聊天工具，而是一套围绕"项目上下文"组织的桌面开发环境。

## 目标用户

- 需要跨多个代码仓库理解项目的开发者
- 需要快速定位代码、文档和历史讨论的团队
- 希望 AI 辅助但保留人工审批控制的工程师
- 关注代码隐私、倾向本地优先的技术团队

## 核心使用场景

### 工作区管理

一个工作区代表一个真实的软件项目，可以包含多个资料源：

```text
电商平台工作区
├─ admin-web                 React 前端仓库
├─ api-server                Rust/Java 后端仓库
├─ deployment                部署仓库
├─ docs                      本地文档目录
├─ GitHub Issues / PR
├─ GitLab Wiki
└─ 工作区笔记
```

### 知识检索

提供四种搜索模式：

- **快速搜索**：类似 IDE 的 `Ctrl + P`，搜索文件名、路径、类名、方法名
- **全文搜索**：搜索代码、注释、文档、提交和 Issue
- **语义搜索**：允许使用自然语言描述查找代码
- **关系搜索**：基于 Code Graph 回答调用关系、依赖关系等问题

### AI 工作台

AI 工作台由多个工作模式组成：

- **项目问答**：回答必须附带本地来源（文件路径、行号、PR）
- **代码分析**：解释文件、生成调用链、排查异常、分析性能
- **架构分析**：自动生成项目结构摘要、识别模块边界、分析循环依赖
- **开发规划**：生成影响范围、修改计划和风险评估（不修改代码）

### 受控变更

当用户要求 AI 修改代码时，创建独立的 Change Session：

```text
AI 请求 → 上下文收集 → 生成修改计划 → 生成 Patch → 变更预览
→ 用户批准 → 写入文件 → 执行验证 → 生成执行报告
```

AI 不拥有直接写文件权限，只有批准后的 ChangeSet 才能交给 Rust 文件执行器。

## 产品价值

1. **更懂整个项目**：跨仓库、跨文档、跨历史的统一知识索引
2. **答案可以验证**：每个关键结论都能定位到具体文件、代码行、Commit、Issue 或 PR
3. **上下文由用户控制**：用户决定发送给模型的内容范围
4. **代码默认留在本地**：本地优先，云端模型需经披露确认
5. **修改前必须审批**：所有文件变更和命令执行都需要人工批准
6. **操作能够回滚**：完整的检查点和回滚机制
7. **历史和当前实现统一检索**：Git 历史、PR、Issue 与当前代码统一搜索
